# Руководство по участию в проекте

Спасибо за интерес к **SprintNova Control**.

Ниже описан рекомендуемый процесс, который помогает вносить изменения предсказуемо и безопасно.

## Перед началом работы
Перед тем как что-то менять:
- прочитайте `README.md`;
- проверьте `CHANGELOG.md`, чтобы понять текущее направление развития;
- убедитесь, что изменение действительно нужно продукту, а не только локально удобно;
- если изменение крупное, сначала согласуйте подход с сопровождающими.

## Как предлагать изменения
Рекомендуемый процесс:
1. Создайте отдельную ветку.
2. Внесите минимально достаточные изменения.
3. Обновите документацию, если поведение проекта изменилось.
4. Проверьте локальный запуск и сборку.
5. Подготовьте понятное описание изменений.

## Локальная разработка
```bash
npm install
npm start
npm run build
```

## Рекомендации по изменениям
### Код
- не смешивайте большой рефакторинг и новую функциональность в одном изменении;
- поддерживайте существующую структуру компонентов, контекста и утилит;
- избегайте дублирования логики;
- сохраняйте UX-последовательность между dashboard, board и admin-частью.

### UI/UX
- новые элементы должны вписываться в текущую дизайн-систему проекта;
- интерфейс должен оставаться понятным без дополнительного обучения;
- если появляется новая сущность или сигнал, она должна быть видна в аналитике и в карточках, если это необходимо.

### Документация
Обновляйте документацию, если вы:
- добавили новую сущность в модель данных;
- изменили правила валидации;
- изменили поведение хранения, импорта/экспорта или фильтрации;
- добавили новую пользовательскую зону или рабочий сценарий.

## Что желательно указывать в описании pull request
- что именно изменено;
- зачем это нужно;
- какие пользовательские сценарии затронуты;
- есть ли миграционные последствия для `localStorage` или JSON-экспорта;
- что было проверено локально.

## Чего лучше избегать
- крупных несогласованных изменений архитектуры;
- смешивания форматирования, переименований и логических правок в одном PR;
- добавления зависимостей без явной причины;
- изменения интерфейса без проверки влияния на уже существующие сценарии.

## Если вы не уверены
Если не уверены в направлении изменений, лучше сначала предложить короткое описание идеи и ожидаемого эффекта, чем присылать большой несогласованный PR.
